# Component Specifications & Testing Contract

> **Note:** This document, along with the rest of the `docs/` directory, is the immutable source of truth for component behaviors, expected inputs/outputs, and edge-case handling. All unit tests must explicitly target a rule defined in this directory. Do not update this document to reflect the code; update the code to reflect this document.

## 1. The Message Bus
* **Initialization:** The Bus must encapsulate private instances of `Inbox` and `Outbox` internally.
* **Inbox Behavior:**
  * Must generate globally unique identifiers for each enqueued `AgentCommand`.
  * Cannot be popped from. The Bus internally tracks claimed command IDs in a private set. Only commands submitted via `broadcast_to_one` participate in the claim lifecycle.
  * Cannot double-claim. For `broadcast_to_one` commands, an already claimed command must return `None` to subsequent `claim()` attempts. For `broadcast_to_all` commands, a command must not be returned more than once to the same `agent_id` across multiple `claim()` calls.
  * `broadcast_to_all` commands are stored in a separate broadcast log inside the Bus. They are never marked as globally claimed. The Bus tracks per-agent delivery by recording which `agent_id` values have received each broadcast. A broadcast will not be returned to the same `agent_id` more than once.
* **Outbox Behavior:**
  * Retains a persistent list of dictionary elements, strictly structured as: `{"request_id": str, "command_name": str, "result": Any, "agent_name": str}`.
  * Must return `None` when `get_result()` queries an unknown ID.

## 2. Agent Commands
* **Data Object Contract:** Must encapsulate a unique string `id`, a string `command_name`, and a dictionary `payload`. AgentCommand carries no claim state — all routing and claim tracking is owned by the Bus.
* **Factory Responsibilities:** `AgentCommandFactory` must contain isolated class/static methods that construct completely valid typed `AgentCommand` objects. The Factory never connects to the Bus.
* **`read_file` Command:** The `read_file` command payload must contain a `path` string. The LLMAgent, upon claiming a `read_file` command it has enqueued, reads the file at that path and writes the raw contents as a string to the Outbox, paired with the original command ID.
* **`toggle_debug_logging` Command:** The payload must strictly contain an `enabled` boolean value. This command must be executed using `broadcast_to_all` so every running agent instance receives it.

## 3. Agent Execution Behavior Constraints
* **Agent Identity:** Every Agent instance must be assigned a globally unique string `id` at construction time. This `id` must be passed to `Bus.claim()` on every call to enable per-agent broadcast delivery tracking.
* **Bus Encapsulation Constraint:** Interacting with the Bus is the sole responsibility of the base Agent. The base Agent must encapsulate its internal reference to the Bus via strong name-mangling (e.g., `self.__bus`) to structurally enforce that derived agents cannot bypass the standard `issue_command` interfaces by accessing the bus directly.
* **Validation Bounds:** Agents must strictly ignore commands in the Bus not present in their explicit `incoming_commands` list.
* **System Initialization:** Base Agents must accept an optional `bootstrap_commands` argument (a list of `AgentCommand` objects) in their `run` method. Prior to entering the polling loop, the agent must submit these commands using the `issue_command()` method, which natively wraps broadcasting and registering the command ID in its internal `waiting_for_results` state tracker.
* **Async Command Dispatch:** The base Agent's `_execute_next_command()` loop must dispatch all valid claims to `_handle_command` via isolated background tasks (e.g., `asyncio.create_task()`) rather than blocking the primary iteration. This guarantees the overarching generic polling loop continuously cycles concurrently.
* **Lifecycle Pipeline:**
  * Repeatedly poll the Bus inbox for matches across their `incoming_commands`.
  * Execute task concurrently without blocking the main loop.
  * Write the result directly back to the Bus Outbox globally from the main loop, guaranteeing that `request_id` maps perfectly to the original claimed `AgentCommand.id`.
* **State Tracking and Resumption:** Agents executing a `_handle_command` task use the `issue_command()` method, which must be declared as an `async` function. The `issue_command()` method natively broadcasts the target command to the Bus, maps it to an `asyncio.Future` managed by the agent, and suspends the current task simply by awaiting that future. Untracked broadcasts must use `issue_broadcast_command()`. When the concurrent `_execute_next_command()` loop eventually polls the tracked result from the Outbox, it officially resolves the mapped future, automatically waking up the suspended handler and returning the final result transparently.
* **Debug Mode Handling:** The Base Agent must implicitly intercept `toggle_debug_logging` broadcasts during its Inbox polling cycle to update its active operational debug state. 
* **Debug Output:** When debug mode is active, the agent must print execution tracing (e.g., claiming assignments or finishing tasks) to the user terminal and append the logs to a timestamped file in a dedicated `logs` directory. A completely new timestamped file must be dynamically created whenever debug logging transitions to `true`.

## 4. LLMAgent Execution Contract
* **LLM Integration:** The `LLMAgent` is explicitly responsible for executing tasks by querying the Gemini model using the `google-genai` SDK and the `GEMINI_API_KEY` environment variable. 
* **Verified Model Constraints:** The `LLMAgent` must use only verified Gemini model identifiers. All valid identifiers must be declared as named class constants on the `LLMAgent` (e.g., `GEMINI_3_FLASH_PREVIEW`, `GEMINI_3_1_PRO_PREVIEW`, `GEMINI_2_5_FLASH`, `GEMINI_2_5_PRO`, `GEMINI_3_1_FLASH_LITE`). Hardcoded model name strings outside of these constant declarations are forbidden.
* **`process_user_prompt` Handler:** When the LLMAgent handles an `AgentCommand` named `process_user_prompt`, it must treat it as a completely isolated, context-free execution. It executes in two phases. Phase 1: The user's prompt is sent to the LLM with an instruction to respond in natural text describing what context it needs before answering. The LLMAgent creates a new `AgentCommand` named `gather_context` with that natural text as its payload, and natively suspends itself by explicitly calling `accumulated_context = await self.issue_command(gather_cmd)`, relying exclusively on the main polling loop to naturally route, claim, and execute the sub-command. Phase 2: Once it wakes up, the LLMAgent prepends the accumulated context to the original prompt and makes a second LLM call. The Phase 2 response is written directly to the Outbox as the final result, with no additional formatting imposed by the agent.
* **`gather_context` Handler:** When the LLMAgent handles an `AgentCommand` named `gather_context`, its payload must contain a `text` string. The LLMAgent calls the LLM instructing it to return a JSON array of `AgentCommand` definitions based on that string. Each `AgentCommand` is processed sequentially inside the handler by safely suspending to query the system using `result = await self.issue_command(cmd)`. The seamlessly returned results are accumulated, concatenated, and written as a single string to the Outbox as the final result, without the agent independently attempting to manually claim or handle the sub-commands inline.

## 5. User Agent Execution Contract
* **Asynchronous REPL Loop:** The `UserAgent` represents a human operator. It inherits the standard asynchronous event loop from the base `Agent`, wrapping terminal reads to ensure they do not block its concurrent background task polling.
* **Symmetrical Integration:** Because it shares the core asynchronous lifecycle, it must adhere identically to the core agent lifecycle by explicitly declaring `incoming_commands` and natively dispatching tasks representing human input via `issue_command()`.
* **Proactive & Reactive Capabilities:** The REPL must proactively accept human input. Terminal inputs `debug on` and `debug off` must automatically trigger a `toggle_debug_logging` dispatch via `issue_broadcast_command()`. System-initiated dialogues (e.g., `prompt_user`) are claimed and handled as standard asynchronous tasks that display the prompt to the terminal and explicitly wait for the human's reply natively.
* **Auto Mode Loop:** When the `UserAgent` handles the `enter_user_auto_mode` command, it processes it as a standard async handler. In this handler, it proactively prompts the user for their initial query and explicitly yields the thread by simply awaiting `self.issue_command()` for a `process_user_prompt` command. When the LLM's final string is returned seamlessly from the await, it displays it to the user and gets input for the next turn. It must provide a mechanism for the user to type "exit" to conclude the auto mode loop. Each round of auto mode is stateless. The human's input for each turn is passed to the LLMAgent as a fresh, isolated prompt with no accumulated conversation history from prior rounds.

## 6. Agent Dialogue Routing & Command Contracts
* **Dialogue Initiation (LLM agent to User agent):** The LLM agent must use the prompt_user command to request input from the human. The payload must strictly contain a `question` string.
* **Dialogue Response (Human to LLM agent):** The User agent handles prompt_user commands. The human reply is not routed as a new command, but must be written directly to the Bus Outbox as a result bound to the original command ID.
* **Passive Dialogue Continuation:** The LLM agent must use its internal waiting_for_results state tracker to asynchronously await the human reply to the prompt_user command in the Outbox before continuing its logic stream.
* **Live Dialogue Routing:** The `UserAgent` exclusively shepherds the auto mode dialogue state. It initiates proactive `process_user_prompt` commands containing the human's text using the `issue_command()` method, which directly routes them to the Message Bus and tracks their Outbox completions. The `LLMAgent` acts purely as an asynchronous executor—it claims `process_user_prompt` commands, coordinates its Phase 1 and 2 operations, and writes the final string to the Outbox. For auto mode conversations, the `LLMAgent` no longer tracks these commands or synthesizes reciprocal `prompt_user` commands in response.
* **System Bootstrapping Validation:** The system entrypoint must start with an empty Message Bus. The dialogue is initiated proactively by the human through the `enter_user_auto_mode` workflow, which seeds the first `process_user_prompt` command into the Bus for the LLMAgent to claim, triggering the automated back-and-forth cycle.
* **Proactive Execution (Human to LLM agent):** The human operator may autonomously instruct the system by instructing the User agent to enqueue a process_user_prompt command. The payload must strictly contain a `prompt` string.
